Screen 02
Invitation Acceptance
CloudPACS
You have been invited
Accept invitation
You've been invited to join Brighton Radiology Clinic as a Radiologist.
  • Invitation-only — no public registrationThis screen is only reachable via an invitation link sent by a client account admin. The URL contains a one-time token: /invite/:token. If the token is invalid or expired, show an error page.
  • Email and role are pre-filled from the invitation tokenThe invitee cannot change their email or role. The account admin set both when sending the invitation. This enforces RBAC from the moment of account creation.
  • FieldsName (free text) + Password + Confirm password. That is all. The email is shown read-only so the user knows which account they are activating.
  • Account name and role displayed prominently"You've been invited to join Brighton Radiology Clinic as a Radiologist." The user must see what they are accepting before setting a password.
  • Token expiry — 48 hoursIf the link has expired, show: "This invitation has expired. Ask your account administrator to send a new invitation." No resend functionality in the internship scope.
  • On submit: activate account, log in, redirect to /patientsNo separate "activation pending" state. Accepting the invitation immediately activates the account and logs the user in.

  • Identical card design to RegisterSame card, same colours, same width. Consistency between auth screens reduces cognitive load.
  • On successStore JWT in localStorage (or httpOnly cookie — discuss with Omer). Redirect to /patients.
  • Error message must be generic"Invalid email or password" — never "User not found" or "Wrong password." Generic errors prevent account enumeration attacks.
  • Forgot passwordOut of scope for the internship build — link can show a "Contact your administrator" tooltip.
  • JWT expiry handlingWhen the token expires and the user makes a request, redirect to /login with a query param: ?reason=session_expired. Show a brief info banner on the login page: "Your session expired. Please sign in again."

  • Search bar is the primary action — full widthPositioned immediately below the page title. Searches by name, MRN, or date of birth. Search triggers on Enter or button click — not on every keystroke (debounced at 300ms for auto-search variant).
  • MUI DataGrid with server-side paginationNever fetch all records. Backend returns 20 rows + totalCount. DataGrid shows correct page count. Sortable columns: Patient Name, Last Study Date.
  • Modality badge on each rowShows the modality of the most recent study. Quick visual scan of the patient list.
  • Click row → Patient DetailNavigates to /patients/:id/studies — the patient's study list. No modal — full-page navigation preserves browser back button.
  • No "New Patient" buttonPatient records are created automatically when a DICOM file is uploaded — the parser reads PatientName, PatientID, and PatientBirthDate from the DICOM header. There is no manual patient creation flow.
  • Empty stateIf search returns no results: centred illustration + "No patients found for '[query]'" + "Clear search" link. Never show a blank DataGrid.

  • Patient header bar — always visibleShows name, MRN, DOB, sex, and study count. A "← All patients" link returns to Patient Search. The header never scrolls away — it is outside the scrollable content area.
  • Two tabs: Studies | Access LogStudies is the default. Access Log is always present — clicking it shows who accessed this patient's records and when. No separate admin screen needed for patient-level audit.
  • Studies table — click "Open" to enter viewerNavigates to /viewer/:studyId. "Open" button is on every row — no need to click the row itself (avoids accidental navigation). Each row shows date, description, modality badge, series count, and image count.
  • Sorted newest firstMost recent study at the top. No user action required — the natural order for clinical review.
  • Patient data is DICOM-derived, read-onlyName, MRN, DOB, and sex come from DICOM tags at upload time. There is no edit button on the patient header — these fields mirror the source DICOM and must not be manually overridden.
  • Access Log — 4 columns: User · Study · Action · TimestampStudy name is its own column — not embedded in the action text. This makes it immediately scannable. Action is a coloured badge: Viewed (blue) or Uploaded (green). More action types can be added later (e.g. Downloaded, AI Report Generated).
  • Study filter dropdownDefault: "All studies" — shows every event for this patient. Selecting a study filters the table to only that study's events. No extra page navigation needed — answers both "who accessed this patient?" and "who accessed this specific study?" from the same screen.
  • Append-only — no delete, no editThe backend never modifies or removes audit rows. The row count at the bottom ("4 events") confirms completeness. Read-only to all users including Account Admins.
  • URL pattern/patients/:id/studies for the Studies tab, /patients/:id/access-log for the Access Log tab. Study filter is a query param: ?study=studyId.

  • Drop zone is the primary interactionLarge, centred, dashed border that turns sky blue on hover. Built with react-dropzone. Accepts any number of .dcm files in one drop — a radiologist may drop thousands at once.
  • After drop: batch summary card — not a file listShow 4 numbers: total files, total size, patients detected, studies detected. These are parsed from DICOM headers client-side. No file names listed — irrelevant to the user and would flood the screen with 1,247 rows.
  • Progress shown per patient — not per fileOne row per patient with a name, study/file/size summary, and a single progress bar. This is the level of granularity the user actually cares about. A per-file list is noise when uploading a large study batch.
  • Overall progress bar in the summary cardAggregate across all patients. Shows overall % and "Uploading…" / "Done" status. The user can see at a glance how far along the batch is.
  • Quota check before upload startsCheck StorageUsedBytes + batchSize > StorageQuota. If over quota, show an error alert above the drop zone and block the upload. Never silently fail mid-batch.
  • On completionShow a MUI Snackbar: "Upload complete — 6 studies indexed across 3 patients." Each patient row shows ✓ Done. No auto-navigation — the user may want to upload another batch.

  • Account Admin sees 3 tabs: Profile & Security · Account · UsersSub-users only see Profile & Security. The Account and Users tabs are hidden entirely — not disabled.
  • Users tab — invitation flow, not registrationAccount admin enters an email + selects a role → clicks "Send invitation" → the invited user receives a link to /invite/:token. No public registration form exists anywhere.
  • 3 roles available to assign: Viewer · Radiologist · UploaderViewer: read-only access to studies. Radiologist: view + generate AI reports. Uploader: upload only (e.g. a scanning technician who does not review images).
  • Invited statusUsers who have been invited but not yet accepted show their email (not name), their assigned role, and an "Invited" status badge. Account admin can resend the invitation.
  • Account tab (not shown)Shows the account name, storage quota bar (used / total), and plan type. Read-only — these values are set by the PostDICOM Console admin, not the client.
  • Account admin cannot change their own rolePrevents accidental self-demotion. Role changes for the account admin must go through PostDICOM Console.
